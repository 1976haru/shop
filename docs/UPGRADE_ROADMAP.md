# Commerce Diagnostic Hub 업그레이드 로드맵

> 기준일: 2026-07-01  
> 저장소: `1976haru/shop`  
> 원칙: 진단·제안은 자동화하되 외부 판매채널의 실제 쓰기는 사람 승인과 안전 게이트 뒤에 둔다.

## 1. 현재 제품 위치

이 프로젝트는 두 경로를 분리한다.

### 경로 A — 등록 전 진단 제품

공급사 파일을 표준화하고 가격·규정·채널 호환성·등록 준비도를 검토하는 로컬 우선 도구다. 판매자 계정이나 채널 API 키 없이도 사용할 수 있다.

### 경로 B — 실제 판매 운영 자동화

쿠팡·스마트스토어 상품 등록, 가격·재고, 주문·송장 처리를 연결하는 운영 시스템이다. 사업자·판매자 계정·채널별 자격과 API 권한이 준비된 뒤에만 연다.

현재 우선순위는 **경로 A를 실사용 가능한 상품으로 완성한 뒤 경로 B를 읽기 전용부터 단계적으로 여는 것**이다.

## 8. 에이전트 설계 원칙

### 에이전트가 해도 되는 일

- 카테고리 후보 추천
- 표현 위험 후보 탐지
- 제목·상세설명 초안
- 가격변경 제안
- CS 답변 초안
- 공개·허용된 데이터원의 읽기 전용 조사

### 에이전트가 단독으로 하면 안 되는 일

- 상품 실제 등록
- 가격·재고 실제 변경
- 주문 확정·취소·환불
- 원산지·인증·효능·식별번호 추측
- 승인되지 않은 크롤링이나 접근제어 우회

### 구조

```text
packages/core
  순수 계산·검증·스키마·진단
  네트워크 호출 금지

apps/agent-worker
  외부 데이터·LLM 호출
  구조화된 Proposal[]만 반환

apps/channel-worker
  채널 API 읽기/쓰기
  dry-run·사람 승인·멱등성·감사 로그 필수
```

에이전트 결과는 사실이 아니라 **제안**이다. 규칙 엔진이 최종 안전 판정을 내리고, 외부 쓰기는 사람 승인 뒤 채널 작업자가 수행한다.

## 9. 개발 순서

1. Phase 1B: XLSX와 컬럼 매핑 프로파일
2. Phase 1C: 농산물 규칙팩
3. Phase 1D: 이미지 ZIP 검사
4. Phase 1E: 일괄 수정·정책 영향 분석
5. Phase 3: 실제 채널 읽기 전용
6. Phase 4: 상품 1건 카나리 쓰기
7. Phase 5: 가격·재고·주문

각 기능은 별도 브랜치와 Pull Request로 개발하고 다음 검사를 모두 통과해야 한다.

```bash
npm ci
npm run typecheck
npm test
npm run check
npm run diagnose -- fixtures/slice0/normal-agri.csv --out tmp/roadmap-check.json
```

## 10. 제품 완료 기준

### 진단 제품 1.0

- 공급사 CSV/XLSX를 별도 수작업 없이 불러온다.
- 매핑 프로파일을 재사용한다.
- 잘못된 행이 전체 실행을 중단하지 않는다.
- 사용자가 오류를 화면에서 수정하고 재진단한다.
- 가격정책 변경 영향을 비교한다.
- 농산물 등록 전 필수 데이터와 증빙을 확인한다.
- 이미지 패키지를 안전하게 검사한다.
- 결과와 수정 이력을 내보낼 수 있다.

### 운영 자동화 1.0

- 실제 채널 상품을 읽어 로컬 표준과 비교한다.
- 모든 쓰기는 dry-run과 사람 승인을 거친다.
- 멱등성·감사 로그·중지 스위치가 있다.
- 가격·재고·주문 실패가 중복 실행으로 이어지지 않는다.
